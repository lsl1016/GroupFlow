# 二、推荐技术选型

## 1. 前端技术选型

| 模块           | 技术                                         |
| -------------- | -------------------------------------------- |
| 前端框架       | React / Vue 均可                             |
| 推荐方案       | React + TypeScript                           |
| UI 组件库      | Ant Design Mobile / NutUI / 自定义移动端组件 |
| 状态管理       | Zustand / Pinia                              |
| 网络请求       | Axios                                        |
| WebSocket 管理 | 原生 WebSocket 封装                          |
| 列表性能       | 虚拟列表                                     |
| 构建工具       | Vite                                         |

如果你主要想练后端高并发，前端不要搞太复杂，推荐：

```text
React + TypeScript + Vite + Zustand + Axios + 原生 WebSocket
```

原因：

1. TypeScript 对消息协议建模更清晰。
2. Zustand 比 Redux 轻，适合聊天状态管理。
3. 原生 WebSocket 更有利于理解底层连接、心跳、重连、消息 ACK。
4. 大群消息列表需要虚拟列表，避免前端渲染卡顿。

------

## 2. 后端技术选型

| 模块      | 推荐技术                             |
| --------- | ------------------------------------ |
| 后端语言  | Go                                   |
| HTTP 框架 | Gin / Hertz                          |
| WebSocket | Gorilla WebSocket / nhooyr websocket |
| ORM       | GORM / sqlx                          |
| 数据库    | MySQL                                |
| 缓存      | Redis                                |
| 消息队列  | Kafka / RocketMQ / NATS              |
| 日志      | Zap / Zerolog                        |
| 配置      | Viper                                |
| 监控      | Prometheus + Grafana                 |
| 链路追踪  | OpenTelemetry                        |
| 部署      | Docker Compose 起步，后续 Kubernetes |

推荐后端组合：

```text
Go + Gin + Gorilla WebSocket + MySQL + Redis + Kafka + Zap + Prometheus
```

原因：

1. Go 适合长连接和高并发。
2. Gin 上手快，适合业务 API。
3. WebSocket 自己封装更利于学习连接管理。
4. Redis 适合存在线状态、连接路由、群最大 sequence、限流数据。
5. Kafka 适合消息异步广播和削峰。
6. MySQL 负责强一致的群、成员、消息数据。
7. Prometheus 方便学习系统指标。

## 3. 数据库选型

### 主数据库：MySQL

用于存储：

```text
用户信息
群信息
群成员
群消息
群公告
加群申请
禁言记录
消息撤回记录
```

原因：

1. 群成员、群权限、群状态需要事务保证。
2. 群消息需要持久化。
3. 初期使用 MySQL 方便学习和排查问题。
4. 后期可以围绕消息表做分库分表。

------

### 缓存：Redis

用于存储：

```text
用户在线状态
用户连接路由
群最大 sequence
群成员缓存
群配置缓存
未读数缓存
限流计数
WebSocket 节点心跳
```

典型 Key：

```text
online:user:{userId} -> serverId
server:{serverId}:users -> online user set
group:{groupId}:max_seq -> 最大消息序号
group:{groupId}:config -> 群配置缓存
group:{groupId}:members -> 群成员缓存
rate_limit:group:{groupId}:user:{userId} -> 发言限流
```

------

### 消息队列：Kafka

用于：

```text
群消息异步投递
大群广播削峰
系统通知投递
消息审计
后续扩展消息搜索索引
```

为什么需要 MQ？

如果用户发送消息后，后端同步广播给所有在线成员，会出现问题：

```text
用户发送消息
  ↓
查群成员
  ↓
查在线状态
  ↓
循环推送几千人
  ↓
接口阻塞
  ↓
发送延迟变高
```

引入 MQ 后：

```text
用户发送消息
  ↓
消息服务校验并落库
  ↓
返回 ACK
  ↓
消息进入 Kafka
  ↓
投递服务异步消费
  ↓
分片推送给在线用户
```

这样发送链路更稳定。

# 四、后端服务模块设计

## 1. 群服务 Group Service

负责群相关业务。

核心职责：

```text
创建群
修改群资料
群公告
群角色管理
成员管理
加群审批
退群
踢人
解散群
禁言
大群模式配置
```

核心接口：

```text
POST   /api/groups
GET    /api/groups
GET    /api/groups/{groupId}
PUT    /api/groups/{groupId}
DELETE /api/groups/{groupId}

GET    /api/groups/{groupId}/members
POST   /api/groups/{groupId}/members/invite
DELETE /api/groups/{groupId}/members/{userId}
POST   /api/groups/{groupId}/leave

POST   /api/groups/{groupId}/join-requests
POST   /api/groups/{groupId}/join-requests/{requestId}/review

POST   /api/groups/{groupId}/mute-all
POST   /api/groups/{groupId}/members/{userId}/mute
```

------

## 2. 消息服务 Message Service

负责群消息发送和存储。

核心职责：

```text
消息内容校验
成员身份校验
禁言校验
生成 messageId
生成 group sequence
消息去重
消息落库
返回 ACK
写入 Kafka
历史消息查询
消息撤回
```

核心接口：

```text
POST /api/groups/{groupId}/messages
GET  /api/groups/{groupId}/messages?beforeSequence=xxx&limit=20
POST /api/groups/{groupId}/messages/{messageId}/recall
POST /api/groups/{groupId}/read
```

------

## 3. WebSocket Gateway

负责长连接。

核心职责：

```text
建立连接
鉴权
心跳检测
断线清理
连接注册
消息接收
消息推送
重连处理
```

连接建立：

```text
ws://host/ws?token=xxx
```

连接成功后：

```text
1. 校验 token
2. 获取 userId
3. 生成 connectionId
4. 注册到本机连接表
5. 注册到 Redis 全局路由
6. 返回连接成功事件
```

------

## 4. 投递服务 Delivery Service

负责群消息广播。

核心职责：

```text
消费 Kafka 消息
查询群成员
查询在线用户
按 WebSocket 节点分组
生成投递任务
推送给对应 WebSocket 节点
失败重试
记录投递指标
```

大群投递时，不应该逐个用户同步推送，而应该按节点拆分：

```text
groupId = 10001
onlineUsers = 3000

server_1: 800
server_2: 700
server_3: 600
server_4: 500
server_5: 400
```

投递服务生成 5 个投递任务：

```text
server_1 -> 800 个本机用户
server_2 -> 700 个本机用户
server_3 -> 600 个本机用户
server_4 -> 500 个本机用户
server_5 -> 400 个本机用户
```

------

## 5. 在线状态服务 Online Service

可以初期内嵌在 WebSocket Gateway 中，后期拆出来。

核心职责：

```text
用户上线
用户下线
多端连接
连接路由
在线人数统计
群在线人数统计
```

Redis 数据结构示例：

```text
user:online:{userId} -> serverId
server:users:{serverId} -> userId set
group:online:{groupId} -> userId set
```

注意：大群不建议频繁精确维护每个群的在线成员集合，可以改成近似统计或定期刷新。

# 七、核心链路设计

## 1. 用户进入群聊

```text
用户点击群聊
  ↓
HTTP 查询群详情
  ↓
HTTP 查询当前用户群成员状态
  ↓
HTTP 查询最近 20 条历史消息
  ↓
WebSocket 保持连接
  ↓
客户端根据 lastReadSequence 计算未读位置
  ↓
展示群聊页面
```

------

## 2. 发送群消息

```text
用户输入消息
  ↓
客户端生成 clientMessageId
  ↓
本地插入 sending 消息
  ↓
通过 WebSocket 发送 group_message_send
  ↓
WebSocket Gateway 接收消息
  ↓
调用 Message Service
  ↓
校验用户是否在群内
  ↓
校验群是否正常
  ↓
校验用户是否被禁言
  ↓
校验慢速模式和限流规则
  ↓
生成 group sequence
  ↓
消息落库
  ↓
返回 ACK 给发送者
  ↓
写入 Kafka
  ↓
Delivery Service 消费
  ↓
查询群内在线用户
  ↓
按 WebSocket 节点分片
  ↓
推送给在线用户
```

------

## 3. 消息补拉

断线重连后：

```text
客户端记录 lastReceivedSequence
  ↓
WebSocket 重连成功
  ↓
客户端请求：
GET /api/groups/{groupId}/messages?afterSequence=xxx
  ↓
服务端返回遗漏消息
  ↓
客户端按 sequence 合并到消息列表
```

如果收到消息乱序：

```text
客户端收到 sequence = 1005
但本地最后一条是 1002
  ↓
发现缺少 1003、1004
  ↓
调用补拉接口
  ↓
补齐消息
  ↓
重新排序展示
```

# 八、大群性能设计

## 1. 大群要避免的设计

不要这样做：

```text
每发一条群消息：
1. 查询所有群成员
2. 给每个成员插入一条收件箱记录
3. 给每个成员计算未读
4. 给每个成员逐个推送
```

这个设计在万人群会很快爆炸。

------

## 2. 推荐设计原则

```text
群消息只存一份
成员只记录 lastReadSequence
在线用户实时推送
离线用户上线后补拉
大群成员分页加载
大群不展示完整已读名单
大群广播异步化
大群推送按节点分片
```

------

## 3. 大群消息广播设计

### 小群

```text
消息服务可以直接广播
```

### 大群

```text
消息服务只负责：
1. 校验
2. 落库
3. ACK
4. 写 Kafka

投递服务负责：
1. 消费消息
2. 查询在线成员
3. 按 serverId 分组
4. 批量推送
```

------

## 4. 大群成员列表

不允许一次性返回全部成员。

接口必须是：

```text
GET /api/groups/{groupId}/members?cursor=xxx&limit=50
```

不要使用深分页：

```text
page=10000&pageSize=50
```

原因：

```text
大群成员几十万时，深分页性能很差
```

------

## 5. 大群历史消息

使用游标分页：

```text
GET /api/groups/{groupId}/messages?beforeSequence=100201&limit=20
```

查询逻辑：

```sql
SELECT *
FROM group_message
WHERE group_id = ?
  AND sequence < ?
ORDER BY sequence DESC
LIMIT 20;
```

依赖索引：

```sql
INDEX idx_group_sequence(group_id, sequence)
```

------

## 6. 未读数设计

不要给每个用户每条消息记录已读。

推荐：

```text
group_member.last_read_sequence
```

未读数：

```text
group_max_sequence - last_read_sequence
```

大群中可以展示：

```text
99+
```

而不是必须展示精确到几万条。

------

## 7. 限流与慢速模式

大群必须提前有慢速模式。

Redis Key：

```text
rate_limit:group:{groupId}:user:{userId}
```

规则：

```text
普通成员每 N 秒只能发送 1 条消息
管理员不受限制或限制较宽
@所有人单独限频
```

# 九、消息可靠性设计

## 1. 发送 ACK

客户端发送消息后，必须等待服务端 ACK。

客户端状态流转：

```text
sending -> success
sending -> failed
failed -> retry
```

------

## 2. 消息幂等

客户端每条消息生成：

```text
clientMessageId
```

服务端唯一约束：

```sql
UNIQUE KEY uk_sender_client_msg(sender_id, client_message_id)
```

如果客户端重复发送，服务端不重复落库，而是返回之前的消息结果。

------

## 3. 消息顺序

每个群维护递增 sequence。

Redis 生成：

```text
INCR group:{groupId}:sequence
```

消息排序：

```text
groupId + sequence
```

不要依赖：

```text
created_at
```

原因：

```text
分布式场景下，不同机器时间可能存在误差。
```

------

## 4. 消息不丢

原则：

```text
先落库，再 ACK，再异步广播
```

如果广播失败，用户仍然可以通过历史消息补拉。

发送链路：

```text
写库成功 -> ACK 成功 -> Kafka 投递
```

如果 Kafka 投递失败，可以：

```text
1. 本地事务表
2. 定时扫描未投递消息
3. 重新投递 Kafka
```

初期可以先简单实现，后期再做可靠事件表。

# 十六、整体方案一句话总结

这个群聊系统的整体技术方案可以概括为：

```text
前端通过 WebSocket 发送和接收群消息；
后端使用 Go 管理长连接、群权限和消息落库；
MySQL 存储群、成员和消息；
Redis 维护在线状态、群 sequence、缓存和限流；
Kafka 承接大群消息异步投递；
消息通过 groupId + sequence 保证顺序；
通过 clientMessageId 保证幂等；
通过 lastReadSequence 实现未读和离线补拉；
大群通过分页、限流、异步广播、分片推送保证性能。
```
