# 群聊系统产品原型文档

## 1. 文档说明

### 1.1 文档目的

本文档用于规划一个多人实时在线群聊系统的产品原型，重点关注群聊完整功能、大群使用场景、消息可靠性、成员管理、权限控制、已读未读、禁言、公告、系统通知、消息撤回、@提醒等能力。

本项目的主要目标不是实现完整社交产品，而是通过群聊系统学习高并发系统设计。因此，产品设计阶段需要提前考虑大群性能问题，避免后期功能设计与技术架构冲突。

------

## 2. 产品定位

### 2.1 产品名称

群聊实时通信系统

### 2.2 产品目标

构建一个支持多人实时群聊的系统，满足以下能力：

1. 支持用户创建群、加入群、退出群。
2. 支持群成员角色管理，包括群主、管理员、普通成员。
3. 支持实时发送和接收群消息。
4. 支持大群场景下的消息广播、未读数、历史消息加载。
5. 支持消息可靠投递，包括发送确认、失败重试、消息去重。
6. 支持群公告、禁言、加群审批、成员管理等完整群管理能力。
7. 支持基础运营管控能力，包括群解散、成员踢出、消息撤回、敏感操作系统通知。

### 2.3 不包含范围

当前阶段不关注以下能力：

1. 单聊。
2. 音视频通话。
3. 朋友圈 / 动态。
4. 红包。
5. 直播。
6. 支付。
7. 小程序。
8. 复杂推荐算法。

------

## 3. 用户角色

### 3.1 系统用户

系统中的用户可以加入多个群聊，也可以创建群聊。

### 3.2 群内角色

群内角色分为三类：

| 角色     | 说明                                 |
| -------- | ------------------------------------ |
| 群主     | 群的创建者或拥有者，拥有最高管理权限 |
| 管理员   | 群主设置的管理人员，拥有部分管理权限 |
| 普通成员 | 普通群成员，主要负责参与聊天         |

### 3.3 角色权限

| 功能           | 群主                   | 管理员 | 普通成员       |
| -------------- | ---------------------- | ------ | -------------- |
| 修改群名称     | 可以                   | 可以   | 不可以         |
| 修改群头像     | 可以                   | 可以   | 不可以         |
| 修改群公告     | 可以                   | 可以   | 不可以         |
| 邀请成员       | 可以                   | 可以   | 根据群配置决定 |
| 审批加群申请   | 可以                   | 可以   | 不可以         |
| 踢出成员       | 可以                   | 可以   | 不可以         |
| 设置管理员     | 可以                   | 不可以 | 不可以         |
| 取消管理员     | 可以                   | 不可以 | 不可以         |
| 解散群聊       | 可以                   | 不可以 | 不可以         |
| 退出群聊       | 不适用，需要转让或解散 | 可以   | 可以           |
| 发送普通消息   | 可以                   | 可以   | 可以           |
| 发送 @所有人   | 可以                   | 可以   | 不可以         |
| 撤回自己的消息 | 可以                   | 可以   | 可以           |
| 撤回他人消息   | 可以                   | 可以   | 不可以         |
| 全员禁言       | 可以                   | 可以   | 不可以         |
| 单人成员禁言   | 可以                   | 可以   | 不可以         |

------

## 4. 群类型设计

### 4.1 普通群

普通群用于中小规模聊天场景。

建议人数范围：

```text
1 - 500 人
```

普通群支持完整功能，包括：

1. 群成员列表。
2. 单条消息已读详情。
3. @某人。
4. @所有人。
5. 群公告。
6. 群成员管理。
7. 消息撤回。
8. 禁言。

### 4.2 大群

大群用于高并发学习场景。

建议人数范围：

```text
500 - 100000 人
```

大群在产品能力上需要做限制，避免系统压力过大。

大群限制规则：

| 能力             | 普通群             | 大群                                   |
| ---------------- | ------------------ | -------------------------------------- |
| 展示完整成员列表 | 支持               | 默认不支持完整加载，改为分页搜索       |
| 单条消息已读详情 | 支持               | 不展示完整已读名单，只展示概览或不展示 |
| @所有人          | 支持               | 严格限频                               |
| 历史消息加载     | 普通分页           | 游标分页，限制单次条数                 |
| 消息发送频率     | 普通限制           | 更严格限制                             |
| 在线成员展示     | 可展示完整在线状态 | 只展示在线人数，不展示全部在线列表     |
| 群消息实时推送   | 全量在线成员推送   | 仅在线成员推送，离线成员上线后拉取     |

------

## 5. 核心页面原型

## 5.1 群聊主页面

### 5.1.1 页面目标

群聊主页面是用户参与群聊的核心页面，包含群列表、当前群消息窗口、群信息侧栏。

### 5.1.2 页面布局

```text
+--------------------------------------------------------------------------------+
| 顶部栏：当前登录用户 / 网络状态 / WebSocket连接状态 / 系统通知入口                |
+----------------------+-----------------------------------+---------------------+
| 群列表                | 群聊消息区                         | 群信息侧栏           |
|                      |                                   |                     |
| 技术交流群            | 群名称：技术交流群                  | 群头像               |
| 未读 12              | 在线人数：128                      | 群名称               |
|                      |                                   | 群公告               |
| 架构设计群            |-----------------------------------| 群成员入口           |
| 有人@我              | 消息列表                           | 群设置               |
|                      |                                   |                     |
| 高并发学习群          | user_001: 大家好                   |                     |
| 未读 99+             | user_002: 收到                     |                     |
|                      |                                   |                     |
|                      |-----------------------------------|                     |
|                      | 输入框                             |                     |
|                      | [@] [表情] [发送]                  |                     |
+----------------------+-----------------------------------+---------------------+
```

### 5.1.3 群列表展示字段

每个群列表项展示：

| 字段             | 说明                              |
| ---------------- | --------------------------------- |
| 群头像           | 群的头像                          |
| 群名称           | 群聊名称                          |
| 最后一条消息摘要 | 展示最近一条消息内容              |
| 最后一条消息时间 | 展示最近消息时间                  |
| 未读数           | 当前用户在该群的未读消息数        |
| @提醒            | 如果有人 @ 当前用户，需要高亮展示 |
| 群状态           | 正常、已解散、已禁言等            |

### 5.1.4 群列表未读展示规则

未读数展示规则：

| 未读数  | 展示         |
| ------- | ------------ |
| 0       | 不展示红点   |
| 1 - 99  | 展示具体数字 |
| 大于 99 | 展示 99+     |

如果用户被 @，优先展示：

```text
[有人@我]
```

如果存在 @所有人，展示：

```text
[@所有人]
```

## 5.2 群聊消息区

### 5.2.1 页面目标

展示当前群的历史消息和实时消息，支持用户发送消息、查看消息状态、处理消息操作。

### 5.2.2 消息区结构

```text
+------------------------------------------------+
| 群名称：高并发学习群                            |
| 群人数：12000    在线：3260                     |
+------------------------------------------------+
| 加载更多历史消息                                |
|                                                |
| user_001 10:01                                 |
| 大家今天讨论 WebSocket 网关设计。               |
|                                                |
| user_002 10:02                                 |
| 收到。                                         |
|                                                |
| 系统消息：管理员开启了全员禁言                  |
|                                                |
| 我 10:03                                       |
| 这个消息正在发送中...                           |
| 状态：发送中                                   |
+------------------------------------------------+
| 输入框                                          |
| [输入消息...]                                  |
| [发送]                                         |
+------------------------------------------------+
```

### 5.2.3 消息展示字段

每条消息展示：

| 字段       | 说明                         |
| ---------- | ---------------------------- |
| 发送人头像 | 消息发送者头像               |
| 发送人昵称 | 群内昵称优先                 |
| 发送时间   | 消息发送时间                 |
| 消息内容   | 文本、系统消息等             |
| 消息状态   | 发送中、已发送、失败、已撤回 |
| 操作入口   | 撤回、复制、引用等           |

### 5.2.4 消息状态

客户端侧消息状态包括：

| 状态     | 说明     |
| -------- | -------- |
| sending  | 正在发送 |
| success  | 发送成功 |
| failed   | 发送失败 |
| recalled | 已撤回   |

状态展示规则：

1. 用户点击发送后，消息先进入 sending 状态。
2. 服务端返回 ACK 后，消息变为 success。
3. 超时未收到 ACK，消息变为 failed。
4. 用户可以点击 failed 消息重新发送。
5. 被撤回的消息展示为“某某撤回了一条消息”。

## 6. 核心功能设计

## 6.1 创建群聊

### 6.1.1 功能说明

用户可以创建群聊，创建者自动成为群主。

### 6.1.2 输入字段

| 字段     | 必填 | 说明                |
| -------- | ---- | ------------------- |
| 群名称   | 是   | 群聊名称            |
| 群头像   | 否   | 群头像              |
| 群简介   | 否   | 群描述              |
| 初始成员 | 否   | 邀请加入的用户      |
| 入群方式 | 是   | 审批加入 / 邀请加入 |
| 群类型   | 是   | 普通群 / 大群       |

### 6.1.3 规则

1. 群名称不能为空。
2. 群名称长度限制为 2 - 30 个字符。
3. 创建者自动成为群主。
4. 创建成功后生成系统消息。
5. 初始成员加入后生成入群系统消息。
6. 群默认状态为正常。

------

## 6.2 加入群聊

### 6.2.1 加入方式

支持三种方式：

| 加入方式 | 说明                         |
| -------- | ---------------------------- |
| 审批加入 | 用户提交申请，管理员审批     |
| 邀请加入 | 群成员邀请用户加入           |
| 直接加入 | 用户可以直接加入，适合开放群 |

### 6.2.2 加群限制

1. 被群拉黑的用户不能加入。
2. 群已解散时不能加入。
3. 群人数达到上限时不能加入。
4. 用户重复提交申请需要限频。
5. 大群加群需要异步处理，避免瞬时大量用户加入造成压力。

------

## 6.3 退出群聊

### 6.3.1 功能说明

普通成员和管理员可以主动退出群聊。

### 6.3.2 规则

1. 群主不能直接退出，需要先转让群主或解散群。
2. 退出后不再接收新消息。
3. 退出后历史消息是否可见由群配置决定。
4. 退出后群内生成系统消息。
5. 退出后该用户的群会话从列表中移除。

------

## 6.4 解散群聊

### 6.4.1 功能说明

群主可以解散群聊。

### 6.4.2 规则

1. 只有群主可以解散群。
2. 解散需要二次确认。
3. 解散后群状态变为 deleted。
4. 解散后所有成员不能发送消息。
5. 解散后历史消息可以保留。
6. 解散后所有在线成员收到系统通知。

------

## 6.5 群消息发送

### 6.5.1 支持的消息类型

初期支持：

| 类型   | 说明                 |
| ------ | -------------------- |
| text   | 文本消息             |
| system | 系统消息             |
| image  | 图片消息，可后续实现 |
| file   | 文件消息，可后续实现 |
| quote  | 引用消息，可后续实现 |

### 6.5.2 文本消息规则

1. 消息内容不能为空。
2. 消息长度限制，例如 1 - 2000 字符。
3. 用户必须是群成员。
4. 用户不能处于被禁言状态。
5. 群不能处于全员禁言状态，除非用户是群主或管理员。
6. 群已解散时不能发送消息。
7. 消息需要服务端生成全局唯一 messageId。
8. 消息需要服务端生成群内递增 sequence。
9. 客户端需要传 clientMessageId，用于消息去重。

### 6.5.3 发送流程

```text
用户输入消息
  ↓
点击发送
  ↓
客户端生成 clientMessageId
  ↓
消息进入发送中状态
  ↓
客户端通过 WebSocket 发送消息
  ↓
服务端校验身份、群成员、禁言、群状态
  ↓
服务端生成 messageId 和 sequence
  ↓
服务端保存消息
  ↓
服务端返回 ACK
  ↓
客户端更新消息状态为发送成功
  ↓
服务端向群内在线成员广播消息
```

### 6.5.4 发送失败处理

失败原因包括：

| 原因       | 客户端展示           |
| ---------- | -------------------- |
| 网络异常   | 发送失败，可重试     |
| 用户被禁言 | 当前处于禁言状态     |
| 群已解散   | 群聊已解散，无法发送 |
| 不是群成员 | 你不是该群成员       |
| 消息过长   | 消息内容过长         |
| 频率过高   | 发送过快，请稍后再试 |

------

## 6.6 群消息接收

### 6.6.1 在线用户

在线用户通过 WebSocket 实时接收群消息。

### 6.6.2 离线用户

离线用户不做实时投递。用户上线后，根据自己的读取位置拉取历史消息。

### 6.6.3 大群接收规则

大群场景下：

1. 只对在线用户做实时推送。
2. 不为离线用户逐个生成离线消息。
3. 用户重新进入群聊后，通过 lastReadSequence 拉取遗漏消息。
4. 如果遗漏消息过多，只拉取最近一段消息，并提示“有更多历史消息”。

------

## 6.7 历史消息

### 6.7.1 功能说明

用户进入群聊时，需要加载历史消息。

### 6.7.2 加载规则

1. 默认加载最近 20 条消息。
2. 向上滚动加载更早消息。
3. 使用游标分页，不使用深分页。
4. 排序依据为 groupId + sequence。
5. 已撤回消息仍展示撤回提示。
6. 已删除消息根据规则决定是否展示。

### 6.7.3 大群历史消息规则

大群必须使用游标分页：

```text
beforeSequence = 100200
limit = 20
```

不能使用：

```text
page = 10000
pageSize = 20
```

原因是深分页在大数据量下性能较差。

## 6.8 消息撤回

### 6.8.1 功能说明

用户可以撤回自己发送的消息，管理员可以撤回群内成员消息。

### 6.8.2 撤回规则

1. 普通成员只能撤回自己的消息。
2. 普通成员只能在指定时间内撤回，例如 2 分钟。
3. 群主和管理员可以撤回任意成员消息。
4. 消息撤回后不物理删除。
5. 撤回后消息状态改为 recalled。
6. 所有在线成员收到撤回事件。
7. 历史消息中展示“某某撤回了一条消息”。

## 6.9 @ 功能

### 6.9.1 @某人

用户可以在群聊中 @ 一个或多个群成员。

规则：

1. 被 @ 的用户在群列表看到“有人@我”。
2. 被 @ 的用户可以快速跳转到对应消息。
3. @对象必须是当前群成员。
4. 大群中 @ 成员时，需要支持成员搜索。

### 6.9.2 @所有人

@所有人用于重要通知。

规则：

1. 群主和管理员可以 @所有人。
2. 普通成员不能 @所有人。
3. 大群中 @所有人需要严格限频。
4. @所有人消息需要在群列表中突出展示。
5. @所有人可以触发额外通知，但不能无限制推送。

### 6.9.3 大群 @所有人限频规则

建议规则：

| 限制项     | 规则                              |
| ---------- | --------------------------------- |
| 单个管理员 | 每小时最多 3 次                   |
| 单个群     | 每小时最多 10 次                  |
| 大群全局   | 可接入系统级限流                  |
| 超限后     | 提示“@所有人过于频繁，请稍后再试” |

## 6.10 已读未读

### 6.10.1 会话级未读数

每个用户在每个群维护一个读取位置：

```text
lastReadSequence
```

未读数计算方式：

```text
群最大 sequence - 用户 lastReadSequence
```

### 6.10.2 单条消息已读

普通群可以支持单条消息已读人数和未读人数。

计算规则：

```text
成员 lastReadSequence >= 消息 sequence，表示该成员已读该消息
```

### 6.10.3 大群已读限制

大群不展示完整已读名单。

大群可选展示：

1. 不展示单条消息已读。
2. 只展示粗略已读人数。
3. 只对管理员展示统计信息。
4. 只展示最近消息的已读概览。

### 6.10.4 已读上报规则

用户满足以下条件时上报已读：

1. 当前正在查看该群。
2. 消息已经进入可视区域。
3. 用户停留超过一定时间。
4. 客户端合并上报，不每条消息单独上报。

客户端上报：

```text
groupId
lastReadSequence
```

## 6.11 群公告

### 6.11.1 功能说明

群主和管理员可以发布群公告。

### 6.11.2 规则

1. 群公告展示在群信息侧栏。
2. 修改公告后生成系统消息。
3. 用户首次进入群时展示最新公告。
4. 大群公告内容可以缓存。
5. 公告不应每次进入群都强制查询数据库。

## 6.12 禁言

### 6.12.1 全员禁言

群主和管理员可以开启全员禁言。

规则：

1. 全员禁言后，普通成员不能发送消息。
2. 群主和管理员仍可发送消息。
3. 开启或关闭全员禁言时生成系统消息。
4. 输入框区域展示禁言提示。

### 6.12.2 单人成员禁言

群主和管理员可以禁言单个成员。

禁言字段：

```text
禁言用户
禁言截止时间
操作人
禁言原因
```

规则：

1. 被禁言用户不能发送消息。
2. 到期后自动恢复。
3. 管理员不能禁言群主。
4. 普通管理员是否能禁言管理员由群规则决定。
5. 禁言操作生成系统消息或仅通知被禁言人，按配置决定。

## 6.13 系统消息

### 6.13.1 功能说明

系统消息用于展示群内关键事件。

### 6.13.2 系统消息类型

| 类型                 | 示例                     |
| -------------------- | ------------------------ |
| group_created        | user001 创建了群聊       |
| member_joined        | user002 加入了群聊       |
| member_left          | user003 退出了群聊       |
| member_kicked        | user004 被管理员移出群聊 |
| group_renamed        | 群名称已修改             |
| announcement_updated | 管理员更新了群公告       |
| mute_all_enabled     | 管理员开启了全员禁言     |
| mute_all_disabled    | 管理员关闭了全员禁言     |
| message_recalled     | user001 撤回了一条消息   |
| group_dismissed      | 群聊已解散               |

### 6.13.3 系统消息规则

1. 系统消息由服务端生成。
2. 系统消息不能由普通用户伪造。
3. 系统消息需要持久化。
4. 系统消息也需要 sequence。
5. 系统消息参与历史消息排序。

## 7. 大群性能相关产品规则

## 7.1 大群模式触发条件

满足以下任一条件，可进入大群模式：

1. 群人数超过 500。
2. 群在线人数超过 300。
3. 群消息发送 QPS 超过阈值。
4. 管理员手动设置为大群。

## 7.2 大群模式下的产品限制

| 功能         | 限制                               |
| ------------ | ---------------------------------- |
| 群成员列表   | 不一次性加载，必须搜索或分页       |
| 在线成员列表 | 不展示完整列表，只展示在线人数     |
| 单条消息已读 | 默认关闭                           |
| @所有人      | 严格限频                           |
| 历史消息     | 游标分页                           |
| 消息发送     | 频率限制更严格                     |
| 消息搜索     | 后期使用独立搜索服务，不直接查主库 |
| 图片 / 文件  | 可限制大小和频率                   |
| 群公告       | 缓存读取                           |
| 群设置       | 管理操作需要二次确认               |

## 7.3 大群消息策略

产品层面规定：

1. 大群消息只对在线用户实时推送。
2. 离线用户不生成单独离线消息。
3. 离线用户上线后通过历史消息补拉。
4. 客户端需要支持消息快速滚动和虚拟列表。
5. 用户长时间离线后，只展示最近未读摘要，不强制加载全部未读消息。
6. 未读数可以展示为 99+，不要求无限精确展示。
7. 热点大群可以开启慢速模式。

## 7.4 慢速模式

### 7.4.1 功能说明

慢速模式用于限制大群消息发送频率。

### 7.4.2 规则

群主或管理员可以设置：

```text
每个普通成员每 N 秒只能发送 1 条消息
```

例如：

```text
每 10 秒 1 条
每 30 秒 1 条
每 60 秒 1 条
```

### 7.4.3 适用场景

1. 大型公开群。
2. 热点事件讨论群。
3. 消息刷屏严重的群。
4. 系统压力较大时的临时保护。

## 8. 消息可靠性产品规则

## 8.1 发送确认

每条用户主动发送的消息都需要服务端确认。

客户端展示：

| 状态     | 说明                                 |
| -------- | ------------------------------------ |
| 发送中   | 消息已在本地展示，但未收到服务端确认 |
| 已发送   | 服务端已确认                         |
| 发送失败 | 服务端拒绝或网络异常                 |
| 可重试   | 用户可以点击重新发送                 |

## 8.2 消息去重

客户端每次发送消息时生成 clientMessageId。

服务端需要保证：

```text
同一个 senderId + clientMessageId 只能生成一条消息
```

如果客户端重试，服务端返回原消息结果。

## 8.3 断线重连

用户断线后：

1. 客户端展示连接中断。
2. 自动重连 WebSocket。
3. 重连成功后拉取断线期间遗漏的消息。
4. 通过 lastReadSequence 或 lastReceivedSequence 判断遗漏范围。

## 8.4 消息补拉

当客户端检测到 sequence 不连续时，需要主动补拉。

例如客户端收到：

```text
1001
1002
1005
```

说明缺少：

```text
1003
1004
```

客户端需要请求服务端补拉缺失消息。

## 9. 群状态设计

群状态包括：

| 状态      | 说明       |
| --------- | ---------- |
| normal    | 正常       |
| muted     | 全员禁言   |
| dismissed | 已解散     |
| banned    | 被系统封禁 |
| archived  | 已归档     |

不同状态下的行为：

| 状态      | 能否发消息 | 能否看历史     | 能否管理成员 |
| --------- | ---------- | -------------- | ------------ |
| normal    | 可以       | 可以           | 可以         |
| muted     | 部分可以   | 可以           | 可以         |
| dismissed | 不可以     | 可以           | 不可以       |
| banned    | 不可以     | 取决于系统策略 | 不可以       |
| archived  | 不可以     | 可以           | 不可以       |

## 10. 异常场景设计

## 10.1 用户不是群成员却发送消息

处理：

1. 服务端拒绝。
2. 返回错误码。
3. 客户端展示“你不是该群成员”。

## 10.2 用户被踢后仍在群页面

处理：

1. 在线时收到被踢通知。
2. 输入框禁用。
3. 页面提示“你已被移出该群”。
4. 不再接收新消息。

## 10.3 群已解散

处理：

1. 所有在线成员收到群解散通知。
2. 输入框禁用。
3. 页面提示“该群已解散”。
4. 历史消息仍可查看。

## 10.4 消息发送成功但 ACK 丢失

处理：

1. 客户端显示发送失败或等待重试。
2. 用户点击重试。
3. 服务端根据 clientMessageId 识别重复消息。
4. 返回原消息结果。
5. 客户端更新为发送成功。

## 10.5 消息乱序

处理：

1. 客户端按 sequence 排序。
2. 检测 sequence 缺口。
3. 缺失时主动补拉。
4. 补拉成功后重新渲染消息列表。

## 11. 关键交互流程

## 11.1 进入群聊流程

```text
用户点击群聊
  ↓
检查本地是否有群基础信息缓存
  ↓
请求群详情
  ↓
请求最近历史消息
  ↓
请求当前用户在群内的成员状态
  ↓
更新 lastReadSequence
  ↓
展示群聊页面
  ↓
接收实时消息
```

## 11.2 发送消息流程

```text
用户输入消息
  ↓
点击发送
  ↓
客户端生成 clientMessageId
  ↓
本地插入发送中消息
  ↓
WebSocket 发送消息
  ↓
服务端返回 ACK
  ↓
客户端用 messageId 和 sequence 更新本地消息
  ↓
其他在线成员收到广播
```

## 11.3 管理员踢人流程

```text
管理员进入成员管理
  ↓
搜索目标成员
  ↓
点击踢出
  ↓
二次确认
  ↓
服务端校验管理员权限
  ↓
更新成员状态为 kicked
  ↓
通知被踢用户
  ↓
群内生成系统消息
```

## 11.4 开启全员禁言流程

```text
管理员进入群设置
  ↓
开启全员禁言
  ↓
二次确认
  ↓
服务端更新群配置
  ↓
群内生成系统消息
  ↓
普通成员输入框变为不可用
```

## 13. 非功能性产品要求

## 13.1 实时性

普通群目标：

```text
消息端到端延迟 P95 < 500ms
```

大群目标：

```text
消息端到端延迟 P95 < 1000ms
```

## 13.2 可用性

1. WebSocket 断开后客户端自动重连。
2. 服务端异常时客户端可以通过历史消息补拉恢复。
3. 消息落库成功后，即使实时推送失败，也不应丢失。
4. 群状态、成员状态、禁言状态必须由服务端最终判断。

## 13.3 可扩展性

1. 群人数需要支持从几十人扩展到万人级。
2. WebSocket 连接服务需要支持横向扩容。
3. 消息投递需要支持异步化。
4. 历史消息查询需要支持按群分片或按时间归档。
5. 在线状态不能只依赖单机内存。

## 13.4 可观测性

需要记录以下指标：

1. 在线连接数。
2. 群消息发送 QPS。
3. 群消息广播延迟。
4. WebSocket 推送成功率。
5. 消息 ACK 延迟。
6. 消息落库耗时。
7. Redis 操作耗时。
8. 大群活跃度。
9. 热点群排行。
10. 发送失败率。

## 15. 产品验收标准

## 15.1 群聊基础验收

1. 用户可以创建群。
2. 用户可以加入群。
3. 用户可以发送消息。
4. 群内在线成员可以实时收到消息。
5. 刷新页面后可以看到历史消息。
6. 消息按 sequence 正确排序。
7. 用户退出群后不能继续收到新消息。
8. 群解散后不能继续发送消息。

## 15.2 消息可靠性验收

1. 发送消息后客户端能收到 ACK。
2. ACK 丢失后重试不会产生重复消息。
3. WebSocket 断线后可以自动重连。
4. 重连后可以补拉遗漏消息。
5. 客户端收到乱序消息时可以按 sequence 排序。
6. 消息发送失败时有明确提示。

## 15.3 群管理验收

1. 群主可以设置管理员。
2. 管理员可以踢出普通成员。
3. 普通成员不能踢人。
4. 群主可以解散群。
5. 管理员不能解散群。
6. 开启全员禁言后，普通成员不能发言。
7. 被单独禁言的成员不能发言。

## 15.4 大群能力验收

1. 大群成员列表支持分页。
2. 大群不一次性加载全部成员。
3. 大群历史消息使用游标分页。
4. 大群 @所有人 有限频。
5. 大群可以只展示在线人数。
6. 大群可以只对在线成员实时推送。
7. 离线用户重新进入后可以通过历史消息补拉。

5. 多端同步。
6. 音视频能力。
7. 红包等非聊天核心能力。

------

## 17. 总结

本产品原型只关注群聊能力，并在初期就考虑大群性能问题。

核心设计原则是：

1. 群消息只存一份，不给每个用户复制一份。
2. 用户通过 lastReadSequence 记录阅读位置。
3. 群消息通过 groupId + sequence 保证顺序。
4. 在线用户实时推送，离线用户上线后补拉。
5. 大群不展示完整成员列表和完整已读详情。
6. 大群必须支持分页、限流、慢速模式和异步投递。
7. 所有群管理动作都要服务端校验权限。
8. 消息发送必须有 ACK、去重和失败重试机制。

该原型适合作为后续技术设计、数据库设计、WebSocket 协议设计、消息队列设计和压测方案的输入。
