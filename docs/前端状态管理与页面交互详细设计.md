# 前端状态管理与页面交互详细设计文档

## 1. 文档说明

### 1.1 文档目的

本文档用于说明 GroupFlow 群聊系统的前端状态管理和页面交互设计，包括页面结构、状态 Store 拆分、WebSocket 管理、消息发送状态、ACK 处理、断线重连、历史消息补拉、虚拟列表、大群性能优化和异常交互处理。

GroupFlow 是一个面向大群与高并发场景设计的实时群聊系统，当前系统只关注群聊，不设计单聊。

### 1.2 前端设计目标

前端设计需要满足以下目标：

1. 支持群列表展示。
2. 支持群聊窗口实时消息展示。
3. 支持群消息发送、ACK、失败重试。
4. 支持 WebSocket 断线重连。
5. 支持历史消息分页加载。
6. 支持断线后消息补拉。
7. 支持消息去重。
8. 支持按 sequence 排序。
9. 支持未读数展示。
10. 支持 @我提醒。
11. 支持群成员管理。
12. 支持群公告、禁言、慢速模式等交互。
13. 支持大群虚拟列表渲染。
14. 支持大群 UI 降级策略。
15. 支持后续图片、文件、引用消息扩展。

# 5. 前端状态管理总览

## 5.1 Store 拆分

| Store           | 职责                                   |
| --------------- | -------------------------------------- |
| userStore       | 当前用户、token、登录状态              |
| groupStore      | 群列表、当前群详情、群设置             |
| messageStore    | 群消息、发送状态、历史分页、补拉状态   |
| memberStore     | 群成员列表、管理员列表、成员搜索       |
| connectionStore | WebSocket 连接状态、重连状态、心跳状态 |
| mentionStore    | @我提醒、未读 @消息                    |
| uiStore         | 当前页面状态、弹窗、抽屉、加载状态     |

------

## 5.2 状态设计原则

### 5.2.1 服务端数据和 UI 状态分离

服务端数据：

```text
群列表
群详情
群消息
群成员
公告
审批记录
```

UI 状态：

```text
当前选中的群
是否打开群设置抽屉
消息输入框内容
当前滚动位置
是否显示新消息提示
是否正在重连
```

两类状态不要混在一起。

------

### 5.2.2 消息状态必须可追踪

一条消息在前端会经历：

```text
sending
success
failed
recalled
```

必须在状态中保存。

------

### 5.2.3 群消息按 groupId 分桶存储

不要把所有群消息放在一个数组里。

推荐：

```ts
messagesByGroupId: Record<number, GroupMessage[]>
```

原因：

1. 切换群聊时数据隔离。
2. 方便按群做分页。
3. 方便按群维护 lastReceivedSequence。
4. 方便处理大群消息缓存。

## 9.2 消息存储方式

推荐按群存储：

```ts
messagesByGroupId = {
  10001: [
```

------

## 9.3 消息排序规则

所有消息按照：

```text
sequence 升序
```

排序。

本地 sending 消息没有 sequence，可以暂时放在消息列表底部。

ACK 返回后，用服务端 sequence 修正位置。

------

## 9.4 消息去重规则

去重字段优先级：

```text
1. messageId
2. clientMessageId
3. localId
```

收到服务端推送消息时：

```text
如果 messageId 已存在
  更新，不新增

如果 clientMessageId 已存在
  合并 ACK 信息，不新增

否则新增消息
```

# 11. WebSocket Client 设计

## 11.1 核心职责

WebSocket Client 负责：

1. 建立连接。
2. 自动重连。
3. 心跳 ping/pong。
4. 发送消息。
5. 接收消息。
6. 根据 type 分发消息。
7. requestId 与 ACK 匹配。
8. 连接关闭处理。
9. 错误处理。

## 11.5 消息分发

```ts
function handleWsMessage(message: WsMessage) {
  switch (message.type) {
    case "connection_connected":
      handleConnected(message);
      break;
    case "pong":
      handlePong(message);
      break;
    case "group_message_ack":
      handleMessageAck(message);
      break;
    case "group_message_failed":
      handleMessageFailed(message);
      break;
    case "group_message_receive":
      handleMessageReceive(message);
      break;
    case "group_message_recalled":
      handleMessageRecalled(message);
      break;
    case "group_member_kicked":
      handleMemberKicked(message);
      break;
    case "group_dismissed":
      handleGroupDismissed(message);
      break;
    case "error":
      handleWsError(message);
      break;
    default:
      console.warn("unknown ws message type:", message.type);
  }
}
```

# 12. 消息发送交互设计

## 12.1 发送文本消息流程

```text
用户输入消息
  ↓
点击发送
  ↓
前端校验内容非空
  ↓
生成 localId
  ↓
生成 clientMessageId
  ↓
生成 requestId
  ↓
本地插入 sending 消息
  ↓
通过 WebSocket 发送 group_message_send
  ↓
启动 ACK 超时计时器
  ↓
收到 ACK 后更新为 success
```

------

## 12.2 本地消息结构

```ts
const localMessage: GroupMessage = {
  localId: generateLocalId(),
  clientMessageId: generateClientMessageId(),
  groupId,
  senderId: currentUser.userId,
  senderName: currentUser.nickname,
  senderAvatar: currentUser.avatar,
  messageType: "text",
  content,
  status: "sending",
  mentionUserIds,
};
```

------

## 12.3 ACK 处理

收到：

```json
{
  "type": "group_message_ack",
  "data": {
    "groupId": 10001,
    "clientMessageId": "client_msg_001",
    "messageId": "msg_100000001",
    "sequence": 100201,
    "status": "success",
    "createdAt": "2026-06-28T10:00:00.000Z"
  }
}
```

前端处理：

```text
根据 clientMessageId 找本地消息
  ↓
写入 messageId
  ↓
写入 sequence
  ↓
写入 createdAt
  ↓
status = success
  ↓
清除 ACK 超时计时器
  ↓
按 sequence 排序
```

------

## 12.4 ACK 超时

推荐超时时间：

```text
5 秒
```

超时后：

```text
status = failed
显示失败图标
显示“点击重试”
```

------

## 12.5 消息重试

用户点击失败消息：

```text
找到原消息
  ↓
复用原 clientMessageId
  ↓
重新生成 requestId
  ↓
status = sending
  ↓
重新发送 group_message_send
```

注意：

```text
重试必须复用原 clientMessageId。
```

原因：

服务端通过 `senderId + clientMessageId` 做幂等。

# 13. 消息接收交互设计

## 13.1 收到群消息

收到：

```json
{
  "type": "group_message_receive",
  "data": {
    "groupId": 10001,
    "messageId": "msg_100000001",
    "senderId": 1002,
    "senderName": "李四",
    "messageType": "text",
    "content": "收到",
    "sequence": 100202,
    "status": "normal",
    "mentionAll": false,
    "mentionUserIds": [1001],
    "createdAt": "2026-06-28T10:00:01.000Z"
  }
}
```

处理流程：

```text
检查 messageId 是否已存在
  ↓
不存在则插入
  ↓
按 sequence 排序
  ↓
检查 sequence 缺口
  ↓
更新群列表 lastMessage
  ↓
处理未读数和 @提醒
```

------

## 13.2 当前群消息处理

如果用户正在当前群聊天窗口：

```text
插入消息
如果当前滚动在底部，则自动滚动到底部
如果当前不在底部，则显示“有新消息”悬浮按钮
延迟上报已读位置
```

------

## 13.3 非当前群消息处理

如果用户不在该群窗口：

```text
更新群列表 lastMessage
unreadCount + 1
如果 mentionUserIds 包含当前用户，则 mentionMe = true
如果 mentionAll = true，则展示 @所有人提示
```

# 14. Sequence 缺口检测与补拉

## 14.1 lastReceivedSequence

每个群维护：

```ts
lastReceivedSequenceByGroupId: Record<number, number>;
```

------

## 14.2 缺口检测

当前：

```text
lastReceivedSequence = 1002
```

收到：

```text
sequence = 1005
```

说明可能缺失：

```text
1003
1004
```

处理：

```text
触发 HTTP 补拉
GET /api/v1/groups/{groupId}/messages?afterSequence=1002&limit=100
```

------

## 14.3 sequence 跳号处理

服务端允许 sequence 跳号。

如果补拉后仍然没有缺失消息：

```text
接受跳号
更新 lastReceivedSequence
不无限重试
```

# 23. 大群虚拟列表设计

## 23.1 为什么需要虚拟列表

大群消息数量大，如果一次性渲染全部消息，会导致：

1. DOM 数量过多。
2. 页面滚动卡顿。
3. 内存占用高。
4. 消息更新慢。

因此必须使用虚拟列表。

------

## 23.2 虚拟列表要求

需要支持：

```text
动态高度消息
向上加载历史消息
滚动位置保持
新消息追加
跳转到指定 sequence
未读分割线
```

------

## 23.3 新消息合并渲染

如果短时间收到大量消息，不要每条消息都 setState。

推荐：

```text
收到消息先进入 pending 队列
每 50ms - 100ms 批量合并到 messageStore
```

------

## 23.4 当前不在群窗口时

如果用户不在群窗口：

```text
不渲染消息列表
只更新群列表 lastMessage 和 unreadCount
```

避免无意义渲染。

# 29. 性能优化设计

## 29.1 消息列表优化

1. 使用虚拟列表。
2. 消息按 groupId 分桶。
3. 当前不在窗口的群不渲染消息。
4. 批量合并 WebSocket 消息。
5. 图片懒加载。

群列表只展示：

```text

```

不要把所有群消息都放在群列表状态里。

```text
  ↓
进入 pendingMessageQueue
  ↓
每 50ms 批量 flush
  ↓
一次性更新 messageStore
```

减少频繁渲染。

# 32. 总结

GroupFlow 前端状态管理与页面交互设计的核心是：

1. 使用 Zustand 按业务域拆分 Store。
2. userStore 管理用户和 token。
3. groupStore 管理群列表和群详情。
4. messageStore 管理群消息、ACK、补拉和排序。
5. connectionStore 管理 WebSocket 连接、心跳和重连。
6. 群消息按 groupId 分桶存储。
7. 客户端发送消息时生成 clientMessageId。
8. 服务端 ACK 返回后更新本地消息状态。
9. 消息失败后复用 clientMessageId 重试。
10. 收到消息后使用 messageId 去重。
11. 所有消息按 sequence 排序。
12. 发现 sequence 缺口后通过 HTTP 补拉。
13. WebSocket 断线后自动重连并补拉当前群消息。
14. 大群消息列表必须使用虚拟列表。
15. 当前不在窗口的群只更新群列表，不渲染消息列表。
16. 未读数通过 maxSequence - lastReadSequence 展示。
17. 大群 UI 需要对已读详情、成员列表、未读数和 @所有人做降级。
18. 前端权限只负责展示，后端必须做最终权限校验。
