# 压测方案设计文档

## 1. 文档说明

### 1.1 文档目的

本文档用于说明 GroupFlow 群聊系统的压测方案，包括压测目标、压测范围、压测环境、压测工具、压测数据准备、核心压测场景、指标采集、瓶颈分析、风险控制和验收标准。

GroupFlow 是一个面向大群与高并发场景设计的实时群聊系统，系统核心能力包括：

1. WebSocket 长连接。
2. 群消息实时发送。
3. 服务端 ACK。
4. Kafka 异步投递。
5. Delivery Service 大群分片推送。
6. Redis 在线状态和连接路由。
7. MySQL 消息持久化。
8. 历史消息补拉。
9. 大群虚拟列表。
10. 限流、慢速模式和热点群保护。

### 1.2 压测目标

压测目标不是单纯追求最大 QPS，而是验证系统在不同规模下的稳定性、延迟、资源消耗和故障恢复能力。

重点验证：

1. 单 WebSocket 节点最大连接数。
2. 多 WebSocket 节点水平扩展能力。
3. 群消息发送 ACK 延迟。
4. 消息落库性能。
5. Redis sequence 和在线路由性能。
6. Kafka 生产和消费延迟。
7. Delivery Service 大群投递延迟。
8. WebSocket 推送成功率。
9. 大群 fanout 能力。
10. 断线重连和消息补拉能力。
11. 慢速模式和热点群保护是否生效。
12. MySQL、Redis、Kafka、WebSocket Gateway 的瓶颈点。

------

## 2. 压测范围

### 2.1 本次压测覆盖范围

本次压测覆盖以下模块：

| 模块                | 是否压测 | 说明                                     |
| ------------------- | -------- | ---------------------------------------- |
| 登录接口            | 是       | 获取 token，为 WS 连接做准备             |
| 群列表接口          | 是       | 验证群列表查询压力                       |
| 历史消息接口        | 是       | 验证 beforeSequence / afterSequence 查询 |
| 已读上报接口        | 是       | 验证 lastReadSequence 更新能力           |
| WebSocket 连接      | 是       | 验证长连接承载能力                       |
| WebSocket 心跳      | 是       | 验证连接保活能力                         |
| 群消息发送          | 是       | 验证发送、ACK、落库                      |
| Kafka 投递          | 是       | 验证削峰和消费延迟                       |
| Delivery Service    | 是       | 验证大群分片推送                         |
| Redis 在线状态      | 是       | 验证连接路由查询                         |
| MySQL group_message | 是       | 验证消息写入和历史分页                   |
| 前端虚拟列表        | 部分     | 通过浏览器侧专项压测验证                 |
| 文件上传            | 否       | 一期非重点                               |
| 消息搜索            | 否       | 一期可暂缓                               |
| 完整已读名单        | 否       | 大群不重点支持                           |

# 4. 压测工具选型

## 4.1 HTTP 压测工具

推荐：

```text
k6
JMeter
wrk
Locust
```

一期推荐：

```text
k6
```

原因：

1. 支持 HTTP。
2. 支持 WebSocket。
3. 脚本可读性好。
4. 支持阶段化压测。
5. 指标输出方便。

------

## 4.2 WebSocket 压测工具

推荐：

```text
k6 WebSocket
自研 Go WebSocket 压测工具
Gatling
Locust + websocket 插件
```

一期推荐：

```text
自研 Go WebSocket 压测工具 + k6
```

原因：

1. Go 更适合模拟大量长连接。
2. 可以更精确控制连接、心跳、发送消息、接收消息。
3. 可以记录 ACK 延迟和接收延迟。
4. 更贴近后端技术栈。

------

## 4.3 Kafka 压测工具

推荐：

```text
kafka-producer-perf-test
kafka-consumer-perf-test
自研 Kafka producer / consumer
```

用于验证 Kafka 单独吞吐能力。

------

## 4.4 系统监控工具

推荐：

```text
Prometheus
Grafana
Loki
Jaeger 或 OpenTelemetry
pprof
MySQL slow log
Redis slowlog
Kafka exporter
Node exporter
```

# 5. 压测环境设计

## 5.1 最小压测环境

适合本地或开发环境。

```text
1 个 API 服务
1 个 WebSocket Gateway
1 个 Message Service
1 个 Delivery Service
1 个 MySQL
1 个 Redis
1 个 Kafka
```

目标：

1. 验证功能链路。
2. 验证压测脚本。
3. 验证指标采集。
4. 不追求极限性能。

------

## 5.2 标准压测环境

适合阶段性性能测试。

```text
2 个 API 服务
3 个 WebSocket Gateway
2 个 Message Service
2 个 Delivery Service
1 个 MySQL
1 个 Redis
3 个 Kafka Broker
1 套 Prometheus + Grafana
```

目标：

1. 验证多 WS 节点连接路由。
2. 验证 Kafka 削峰。
3. 验证 Delivery Service 水平扩展。
4. 验证大群分片推送。

------

## 5.3 高压压测环境

适合大群高并发验证。

```text
3 - 5 个 API 服务
5 - 10 个 WebSocket Gateway
3 - 5 个 Message Service
5 - 10 个 Delivery Service
1 个 MySQL 主库
1 个 Redis Cluster
3 - 5 个 Kafka Broker
Prometheus + Grafana + Loki + Jaeger
```

目标：

1. 验证 10 万连接。
2. 验证万人在线大群。
3. 验证热点群保护。
4. 验证单点瓶颈。

# 9. 场景 S2：WebSocket 连接压测

## 9.1 目标

验证 WebSocket Gateway 的最大连接数和连接稳定性。

## 9.2 压测方式

模拟用户建立 WebSocket 长连接。

连接后保持在线，不发送高频消息，只维持心跳。

## 9.3 阶段设计

| 阶段   | 连接数 | 持续时间 |
| ------ | ------ | -------- |
| 阶段 1 | 1000   | 10 分钟  |
| 阶段 2 | 5000   | 10 分钟  |
| 阶段 3 | 10000  | 20 分钟  |
| 阶段 4 | 50000  | 30 分钟  |
| 阶段 5 | 100000 | 60 分钟  |

## 9.4 观察指标

1. CPU 使用率。
2. 内存使用率。
3. goroutine 数量。
4. fd 使用数量。
5. 连接成功率。
6. 连接断开率。
7. 心跳超时数。
8. Redis 在线状态写入耗时。

## 9.5 验收标准

| 指标                   | 目标             |
| ---------------------- | ---------------- |
| 连接成功率             | > 99.5%          |
| 非预期断开率           | < 0.5%           |
| 心跳超时率             | < 0.5%           |
| 单 WS 节点内存增长     | 稳定，无持续泄漏 |
| Redis 在线状态写入 P95 | < 50ms           |

------

# 14. 场景 S7：历史消息查询压测

## 14.1 目标

验证历史消息游标分页能力。

## 14.2 压测接口

```text
GET /api/v1/groups/{groupId}/messages?limit=20
GET /api/v1/groups/{groupId}/messages?beforeSequence=100201&limit=20
GET /api/v1/groups/{groupId}/messages?afterSequence=100201&limit=100
```

# 22. 常见瓶颈与优化方向

## 22.1 WebSocket Gateway 瓶颈

可能表现：

1. CPU 高。
2. 内存高。
3. goroutine 数量异常。
4. SendChan 堆积。
5. 推送失败率升高。
6. 心跳超时增加。

优化方向：

1. 增加 WebSocket 节点。
2. 优化 SendChan 大小。
3. 慢连接断开。
4. 批量推送。
5. 降低非核心事件推送。
6. 使用 pprof 分析热点。

------

## 22.2 Message Service 瓶颈

可能表现：

1. ACK 延迟升高。
2. MySQL 写入耗时高。
3. Redis sequence INCR 慢。
4. 权限校验查询慢。
5. 限流判断慢。

优化方向：

1. 缓存群配置。
2. 缓存成员角色。
3. 优化 MySQL 索引。
4. 调整数据库连接池。
5. 引入 Outbox 降低同步 Kafka 影响。
6. 控制消息内容大小。

------

## 22.3 Delivery Service 瓶颈

可能表现：

1. Kafka lag 增长。
2. fanout 处理耗时高。
3. Redis MGET 慢。
4. 内部推送请求堆积。
5. 单节点 CPU 高。

优化方向：

1. 增加 Delivery Service 实例。
2. 增加 Kafka 分区数。
3. 维护群在线成员集合。
4. 按 serverId 维护群在线用户集合。
5. 调整 batchSize。
6. 投递任务并行化。
7. 热点群降级。

------

## 22.4 Redis 瓶颈

可能表现：

1. Redis QPS 高。
2. MGET 延迟高。
3. 大 Key。
4. 内存上涨。
5. slowlog 增加。

优化方向：

1. Pipeline。
2. 减少每条消息 MGET 全成员。
3. 使用 group online users。
4. 使用 server 分片集合。
5. 避免 SMEMBERS 超大集合。
6. 使用 Redis Cluster。
7. 热点 Key 拆分。

------

## 22.5 MySQL 瓶颈

可能表现：

1. group_message 插入慢。
2. 历史消息查询慢。
3. group_member 查询慢。
4. 慢查询增加。
5. 连接池耗尽。

优化方向：

1. 确认 `idx_group_sequence` 命中。
2. 确认 `idx_group_status_id` 命中。
3. 控制单次 limit。
4. 消息表分表。
5. 读写分离。
6. 调整连接池。
7. 批量写入非核心日志。

------

## 22.6 Kafka 瓶颈

可能表现：

1. produce latency 高。
2. consume lag 增长。
3. broker CPU 高。
4. 网络 IO 高。
5. 单分区热点。

优化方向：

1. 增加分区数。
2. 调整 producer batch。
3. 使用压缩。
4. 增加 consumer 实例。
5. 热点群独立 Topic。
6. 慢速模式降低热点群 QPS。

# 23. 压测风险控制

## 23.1 环境隔离

压测必须在独立环境执行，不能影响开发、测试或生产环境。

## 23.2 数据隔离

压测数据使用独立库或独立 schema。

## 23.3 流量逐步增加

不能直接打满最大压力。

必须分阶段递增：

```text
10% -> 30% -> 50% -> 80% -> 100%
```

## 23.4 熔断条件

出现以下情况立即停止压测：

1. MySQL CPU 持续 > 90%。
2. Redis 延迟持续 > 100ms。
3. Kafka lag 快速不可控增长。
4. WebSocket 节点大量崩溃。
5. 错误率 > 5%。
6. 服务出现数据一致性异常。
7. 机器负载持续过高。

# 28. 总结

GroupFlow 压测方案的核心是：

1. 先压 HTTP 基础接口，再压 WebSocket 长连接。
2. 先压小群消息，再压大群 fanout。
3. 先验证单模块，再验证端到端链路。
4. 压测必须覆盖 ACK、落库、Kafka、Delivery、Redis 路由、WebSocket 推送和客户端补拉。
5. 大群压测重点不是所有用户都强一致实时收到，而是消息已落库、在线用户尽量实时收到、失败后可以通过补拉恢复。
6. WebSocket 连接压测重点关注连接数、心跳、内存和文件描述符。
7. 大群投递压测重点关注 fanout、Kafka lag、Delivery 延迟、Redis 路由查询和 WS 推送失败率。
8. 历史消息压测重点验证 `groupId + sequence` 索引。
9. 断线重连压测重点验证最终可见性。
10. 热点群压测重点验证慢速模式、限流和降级策略。
11. 压测必须配套 Prometheus、Grafana、日志、慢查询、Redis slowlog 和 Kafka lag 监控。
12. 压测结果要形成报告，并根据瓶颈逐轮优化。
